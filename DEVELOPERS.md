# Developer Guide

## Цель документа
Правила разработки и поддержки сервиса уведомлений.

## Архитектура
- `src/Controller` — HTTP-эндпоинты.
- `src/Service` — бизнес-логика.
- `src/EventSubscriber/ApiAuthSubscriber.php` — Bearer-аутентификация `/api/*`.
- `migrations/` — миграции БД.
- `docs/openapi.yaml` — контракт API.
- `tests/Api` — интеграционные тесты.

## Технические стандарты
- PHP 8.1+ (фактически 8.3 в контейнере).
- `declare(strict_types=1);` во всех PHP-файлах.
- PSR-12.
- Любая новая бизнес-логика должна покрываться тестами.
- Любые изменения API должны сопровождаться обновлением `docs/openapi.yaml`.

## Принципы кода
- Контроллеры: только orchestration (валидация + вызов сервисов + HTTP-ответ).
- Сервисы: вся предметная логика и транзакции.
- Не дублировать SQL; общие запросы выносить в сервисы/репозитории.
- Не использовать глобальное состояние.
- Исключения наружу только осознанно, с логированием критичных ошибок.

## Правила для БД
- Все изменения схемы только через миграции.
- Имена таблиц/индексов должны быть явными и читаемыми.
- Для hot-path запросов проверять наличие индексов.
- Для сценариев чтения+обновления (unread/read) использовать транзакцию.

## Правила для API
- JSON only.
- Ошибки должны иметь единый формат: `{"error":"..."}`.
- Токен передается только через `Authorization: Bearer <token>`.
- Ограничения валидации должны совпадать с OpenAPI и тестами.

## Тестирование
- Подготовка БД: `make prepare-test`.
- Запуск тестов: `make test`.
- Test DB: `notifications_test`.
- Минимум перед merge:
  - автотесты зеленые,
  - миграции применяются,
  - OpenAPI актуален,
  - README/DEVELOPERS обновлены при изменении поведения.

## Локальный workflow
1. Поднять окружение: `make up`.
2. Внести изменения.
3. Если есть изменения схемы — добавить миграцию.
4. Прогнать `make prepare-test && make test`.
5. Проверить ручные сценарии через curl/HTTP client.

## Логирование и наблюдаемость
- Ошибки пишутся в stderr (Monolog).
- Важные бизнес-события (регистрация пользователя, отправка сообщения, сбои auth) рекомендуется логировать на уровне `info/warning/error`.

## Запрещено
- Хардкодить секреты в коде.
- Менять схему БД без миграции.
- Ломать контракт API без обновления `docs/openapi.yaml`.
- Мержить изменения без автотестов.
